RANDOMNESS ISSUES IN THE DETECTION SYSTEM — EXPLAINED
================================================================

================================================================
ISSUE 1: EPHEMERAL PORTS — WHY traffic.py GENERATES MANY FIRST_SEEN
================================================================

WHAT IS AN EPHEMERAL PORT?
---------------------------
Every TCP connection has TWO endpoints:
    Client side:  IP + PORT  (e.g. h2:54231)
    Server side:  IP + PORT  (e.g. h0:80)

The server's port (80) is fixed — that's how clients know where to connect.
But the CLIENT side also needs a port number. This is automatically assigned
by the OS kernel from a pool of available ports (usually 32768–60999).
This auto-assigned port is called an EPHEMERAL port (ephemeral = temporary).

It is random because:
    - If h2 opens 60 connections to h0:80 at the same time, they all need
      DIFFERENT source ports so h0 knows which reply belongs to which connection.
    - The OS just picks a fresh unused one each time.
    - It changes every connection even if connections are sequential,
      because the OS avoids reusing recently closed ports (safety measure).

THE PROBLEM THIS CAUSES:
-------------------------
Our CMS (Count-Min Sketch) uses the full 5-tuple as the hash key:
    (src_ip, dst_ip, src_port, dst_port, protocol)

With random ephemeral ports:
    Connection 1: (h2, h0, 54231, 80, TCP) → hashes to CMS bucket A
    Connection 2: (h2, h0, 54232, 80, TCP) → hashes to CMS bucket B
    Connection 3: (h2, h0, 54233, 80, TCP) → hashes to CMS bucket C

Each connection hits a DIFFERENT bucket. Each bucket starts at 0.
Bucket at 0 → FIRST_SEEN fires. So 60 connections = up to 60 FIRST_SEEN.
No bucket ever reaches 64 → THRESHOLD never fires.

IS THIS A PROBLEM?
------------------
NO. This is actually CORRECT behavior for legitimate traffic:
    - FIRST_SEEN fires → controller records flow start timestamp (harmless)
    - THRESHOLD never fires → ML never runs → no false positive possible

The system is WORKING correctly. Legitimate random-port traffic
cannot be mistakenly classified as an attack.

DOES IPv4 HAVE THE SAME RANDOMNESS?
-------------------------------------
YES. Ephemeral ports are a TCP feature, not an IPv6 feature.
IPv4 socket.connect() assigns the same random source port.
This is NOT an IPv6 problem. Switching to IPv4 would NOT fix it.

================================================================
ISSUE 2: THE ACK DECREMENT PROBLEM — WHY LEGIT TRAFFIC CAN'T HIT 64
================================================================

THE CMS LOGIC IN P4:
---------------------
SYN (ACK=0):
    → read counter from CMS
    → if counter == 0: send FIRST_SEEN digest
    → increment counter by 1

Any packet with ACK=1 (pure ACK, SYN-ACK, PSH-ACK, FIN-ACK):
    → read counter from CMS
    → if counter > 0: decrement counter by 1

A FULL TCP 3-WAY HANDSHAKE THROUGH THE SWITCH:
------------------------------------------------
Step 1: h2 → h0: SYN (sport=54231)
    CMS bucket for (h2, h0, 54231, 80, TCP): counter 0 → 1
    FIRST_SEEN fired ✓

Step 2: h0 → h2: SYN-ACK (sport=80, dport=54231)
    This has reversed 5-tuple: (h0, h2, 80, 54231, TCP)
    → hashes to a DIFFERENT bucket → decrements that bucket
    (no effect on h2's bucket)

Step 3: h2 → h0: ACK (sport=54231)   ← 3rd step of handshake
    Same 5-tuple as original SYN: (h2, h0, 54231, 80, TCP)
    → same bucket → counter 1 → 0
    COUNTER RESET TO ZERO.

Step 4: Data packets (PSH+ACK), FIN+ACK:
    All have ACK=1 → keep hitting the decrement branch
    Counter is already 0 → no-op (floor at 0)

NET RESULT: Counter goes 0 → 1 → 0. Never reaches 64.

WHY THIS DESIGN IS CORRECT:
-----------------------------
The CMS is designed to count HALF-OPEN connections (SYNs with no matching ACK).
A real SYN flood: attacker sends SYNs but never sends the 3rd ACK.
→ Counter keeps going up: 1, 2, 3 ... 64 → THRESHOLD fires → ATTACK detected.

Legitimate traffic: every SYN is followed by its ACK (handshake completes).
→ Counter resets to 0 after each connection → THRESHOLD never fires → safe.

WHAT THIS MEANS FOR DEMONSTRATION:
-------------------------------------
You CANNOT show BENIGN ML classification using real kernel TCP connections
(traffic.py), because real TCP always resets the counter.

To demonstrate BENIGN classification, you need a script that:
    1. Uses a FIXED source port (same CMS bucket every time)
    2. Sends raw SYNs via Scapy (bypasses kernel → no ACK ever sent)
    3. Runs at LOW rate so ML sees low pps → votes BENIGN
    4. Blocks outgoing RSTs (otherwise h0's SYN-ACK triggers kernel RST
       which decrements the counter — same problem as attack.py had)

This is exactly what benign_flood.py does.

================================================================
ISSUE 3: THE RST PROBLEM — WHY attack.py USED TO GENERATE 421 FIRST_SEEN
================================================================

WHAT HAPPENED:
---------------
After we enabled IPv6 on all host interfaces (sysctl fix), h1's kernel
could now receive and process the SYN-ACK replies from h0.

attack.py sends raw Scapy SYNs. The kernel has no record of these connections.
When the SYN-ACK from h0 arrives at h1:
    h1 kernel sees unexpected SYN-ACK → sends RST-ACK back to h0

That RST-ACK has: src=h1, dst=h0, sport=10000, dport=80, flags=RST+ACK

ACK bit is set → P4 ACK branch → decrements CMS bucket for (h1,h0,10000,80,TCP)

So: SYN → counter=1, RST-ACK arrives → counter=0, next SYN sees 0 → FIRST_SEEN again.
This repeated 200 times → 200 FIRST_SEEN from attack traffic alone.

WHY DIDN'T THIS HAPPEN IN THE FIRST SUCCESSFUL RUN?
------------------------------------------------------
In the first working test, IPv6 was DISABLED on h1's interface.
The SYN-ACK from h0 arrived at h1 but the kernel dropped it silently
(IPv6 disabled → kernel ignores IPv6 packets on that interface).
No RST was sent. Counter accumulated freely.

After we enabled IPv6 globally (sysctl fix), the RST problem appeared.

THE FIX:
---------
Add an ip6tables rule in attack.py to DROP outgoing RSTs before they leave:

    ip6tables -A OUTPUT -p tcp --tcp-flags RST RST -j DROP

This simulates a realistic SYN flood where the attacker either:
    a) Spoofs the source IP (so RSTs go to a non-existent host)
    b) Has a firewall that drops RSTs
    c) Uses a custom kernel that doesn't respond to unexpected SYN-ACKs

With RSTs blocked:
    SYN → counter=1, SYN-ACK arrives at h1, kernel tries to RST but
    ip6tables drops it → counter stays at 1 → next SYN → counter=2 ...
    → reaches 64 → THRESHOLD → ML votes ATTACK → h1 blocked. ✓
